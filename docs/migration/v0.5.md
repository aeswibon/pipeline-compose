# Migration guide — v0.5 to 1.0

**Target removal:** deprecated patterns stop working in **v1.0.0** (Cross-Repo Orchestrator GA).

v0.5.0 adds **warnings** in `pipeline-compose validate` (with `--repo-root`). With `--strict`, warnings become errors — use this in CI to enforce migration before 1.0.

```bash
pnpm run validate .github/pipelines/pipeline.yml --repo-root . --workflows --strict
```

## Deprecation codes

| Code | What it means | Fix before 1.0 |
|------|----------------|------------------|
| `export.manual-upload-deprecated` | Stage uses `jq` + `upload-artifact` for `outputs.json` | Use [pipeline-compose-export](https://github.com/aeswibon/pipeline-compose-export) |
| `export.missing` | Pipeline declares `outputs:` but stage workflow has no export step | Add export action (or manual upload — deprecated) |
| `uses.monorepo-subpath-deprecated` | `aeswibon/pipeline-compose/run@…` style paths | Use `aeswibon/pipeline-compose-run@…` |
| `uses.master-pin-deprecated` | Action pinned at `@master` | Pin semver tag, e.g. `@v0.4.3` |
| `pipeline.v1-deprecated` | Pipeline file uses `version: 1` with top-level `stages:` | Migrate to `version: 2` with `pipelines:` map |

## 1. Manual artifact export → export action

**Before (deprecated):**

```yaml
- run: |
    mkdir -p pipeline-compose
    jq -n --arg version "$VERSION" '{version: $version}' > pipeline-compose/outputs.json
- uses: actions/upload-artifact@v7
  with:
    name: pipeline-compose-version-sync
    path: pipeline-compose/outputs.json
```

**After:**

```yaml
- uses: aeswibon/pipeline-compose-export@v0.4.3
  if: success()
  with:
    stage_id: version-sync
    outputs: '{"version":"${{ steps.meta.outputs.version }}"}'
```

Contract unchanged: artifact `pipeline-compose-<stage-id>` containing `outputs.json`.

## 2. Monorepo action paths → separate repos

**Before:**

```yaml
uses: aeswibon/pipeline-compose/run@v0.1.0
```

**After:**

```yaml
uses: aeswibon/pipeline-compose-run@v0.4.3
```

| Legacy | Replacement |
|--------|-------------|
| `pipeline-compose/run` | [pipeline-compose-run](https://github.com/aeswibon/pipeline-compose-run) |
| `pipeline-compose/compile` | [pipeline-compose-compile](https://github.com/aeswibon/pipeline-compose-compile) |
| `pipeline-compose/eval` | [pipeline-compose-eval](https://github.com/aeswibon/pipeline-compose-eval) |
| `pipeline-compose/export` | [pipeline-compose-export](https://github.com/aeswibon/pipeline-compose-export) |

## 3. Pipeline schema v1 → v2

**Before (v1):**

```yaml
name: release
version: 1
stages:
  - id: ci
    workflow: .github/workflows/ci.yml
```

**After (v2):**

```yaml
version: 2
companion_workflows:
  - .github/workflows/release.yml
pipelines:
  release:
    stages:
      - id: ci
        workflow: .github/workflows/ci.yml
```

v1 files loaded via `pipeline_dir` remain supported in 0.5 but are deprecated. Prefer one v2 file with `pipeline_file`.

## 4. Semver pins

Replace `@master` and floating branch pins with release tags on each action repo:

```yaml
uses: aeswibon/pipeline-compose-run@v0.4.3
```

## 5. context-merge vs run

If you use **pipeline-compose-run**, you do **not** need **pipeline-compose-context-merge** — run builds `context` from export artifacts automatically. Context-merge remains for custom single-workflow setups without run.

## CI checklist before upgrading to 1.0

- [ ] `validate --strict --workflows` passes with no deprecation codes
- [ ] All stage workflows with `outputs:` use **pipeline-compose-export**
- [ ] Entry workflow uses **pipeline-compose-run** with semver pins
- [ ] Pipeline file is **version: 2** (or you accept v1 removal in 1.0)
- [ ] Cross-repo stages have `repo_tokens_json` documented
