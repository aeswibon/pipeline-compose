# Migration guide — 1.0 GA

**pipeline-compose v1.0.0** is the first stable release. Deprecated patterns from 0.5.x are **removed**, not warned.

If you still run 0.5.x, complete [v0.5 → 1.0 checklist](v0.5.md) first, then upgrade action pins to `@v1.0.0`.

## Breaking changes

| Removed in 1.0 | Required instead |
|----------------|------------------|
| Pipeline schema **v1** (`name`, `version: 1`, top-level `stages:`) | **v2** (`version: 2`, `pipelines:` map) — see [pipeline-v2.schema.json](../../packages/core/schema/pipeline-v2.schema.json) |
| `aeswibon/pipeline-compose/<action>@…` monorepo paths | Separate repos, e.g. `aeswibon/pipeline-compose-run@v1.0.0` |
| `@master` / floating branch pins | Semver tags on each action repo |
| Manual `jq` + `upload-artifact` for stage outputs | [pipeline-compose-export](https://github.com/aeswibon/pipeline-compose-export) |

## Validate before upgrading

```bash
pnpm run validate .github/pipelines/pipeline.yml --repo-root . --workflows --strict
```

Under 1.0, legacy patterns are **errors** (no separate deprecation pass). With `--strict`, workflow hygiene warnings (orphans, group paths) also fail.

## Minimal v2 pipeline

```yaml
version: 2
companion_workflows:
  - .github/workflows/release.yml
pipelines:
  release:
    stages:
      - id: ci
        workflow: .github/workflows/ci.yml
```

Optional: `groups`, multi-pipeline `needs`, `companion_workflows`, cross-repo `repo:` on stages — unchanged from 0.5.

## Action pins (1.0.0)

```yaml
uses: aeswibon/pipeline-compose-run@v1.0.0
uses: aeswibon/pipeline-compose-export@v1.0.0
uses: aeswibon/pipeline-compose-compile@v1.0.0
uses: aeswibon/pipeline-compose-eval@v1.0.0
uses: aeswibon/pipeline-compose-context-merge@v1.0.0
```

## CLI / init

- `pipeline-compose init` writes **v2** pipeline files.
- `pipeline-compose compile` defaults compile-check pin to `@v1.0.0`.

## What did not change

- Export artifact contract: `pipeline-compose-<stage_id>` / `outputs.json`
- Run action context merge from export artifacts
- Cross-repo `repo_tokens_json` behavior
- `validate --mermaid` topology diagrams (including error/blocked node styling)

## Release tags

Tag the monorepo `v1.0.0` to publish action packages and sync workspace versions via CI (see [development.md](../development.md)).
